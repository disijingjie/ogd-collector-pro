## 服务器部署与缓存问题排查（12:30-13:05）

### 问题
用户反馈首页和采集中心仍显示22/23、22省+7种采集策略、22个成功采集+1个待确认。

### 根因诊断
1. **本地Flask**：`debug=False`，模板缓存不刷新，修改后必须重启才能生效
2. **GitHub推送失败**：远程地址是HTTPS，且没有配置SSH key
3. **服务器代码落后**：本地领先服务器6个commit，但推送失败导致服务器未更新
4. **Nginx缓存**：即使服务器文件修改，Nginx可能缓存了旧响应

### 解决过程
1. 生成GitHub专用SSH key (`id_ed25519_github`)
2. 配置 `~/.ssh/config`：GitHub用新key，腾讯云用旧key
3. 用户手动添加公钥到GitHub Settings → SSH Keys
4. 直接SSH到服务器用sed修改模板文件（绕过GitHub）
5. 重载Nginx清除缓存：`sudo systemctl reload nginx`
6. 重写"数据规模总览"模块HTML结构（加 `data-v="2"` 属性，改 `style` 为 `class="stat-note"`），强制浏览器和Nginx识别为新内容

### 修改文件
- `templates/v3_dashboard.html`：重写数据规模总览模块（data-v="2" + stat-note class）
- `templates/v3_collection.html`：批次统计22→23，成功率95.7%→100%
- `app.py`：`debug=False` → `debug=True`（本地开发自动重载）
- `~/.ssh/config`：新增GitHub和腾讯云主机配置

### 关键教训
- 本地开发必须开 `debug=True`，否则模板修改不生效
- 服务器部署优先用SSH直接修改，比GitHub推送更可靠
- Nginx缓存可能导致文件已改但页面仍显示旧内容，需重载或改HTML结构
- Windows PowerShell的ssh-agent服务默认禁用，需手动启动或改用SSH config指定密钥
